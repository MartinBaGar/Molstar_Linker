#+TITLE: Architecture & Execution Flow
#+WEIGHT: 20

* Core Philosophy

Mol* Linker operates across several security boundaries enforced by modern browsers. It uses a *4-Layer Architecture* where each layer has a clearly defined responsibility, ensuring security, performance, and compatibility.

** Key Design Decisions

- *Native Mol* Rendering*: The extension now uses Mol*'s native rendering engine instead of MolViewSpec (MVS), providing better performance and direct access to Mol*'s features.
- *Sandboxed Execution*: Mol* runs in a sandboxed iframe to allow dynamic WebGL shader compilation while maintaining security.
- *Dynamic Permissions*: Custom domains are authorized dynamically, enabling seamless integration with self-hosted instances.

* The 4-Layer Architecture

** 1. Reconnaissance (Content Script)
- *Source*: =src/content.ts=
- *Output*: =content.js= (bundled standalone)
- *Environment*: The host webpage (GitHub, GitLab, custom domains)
- *Role*: Scans the DOM for links to supported structure files using a 3-ring scanner (URL, HTML attributes, surrounding text).
- *Action*: Injects a *Mol** badge next to each valid link. Clicking the badge sends an =open_viewer= message to the background script.

** 2. The Router (Background Script)
- *Source*: =src/background.ts=
- *Output*: =background.js=
- *Environment*: Extension service worker (Chrome) or event page (Firefox)
- *Role*: Acts as a security checkpoint and traffic router.
- *Actions*:
  1. Validates =open_viewer= messages (HTTPS URLs, known formats).
  2. Opens a new tab with =viewer.html=, passing the file URL and format as query parameters.
  3. Handles the right-click context menu (*"Open in Mol* Workspace"*).
  4. Dynamically registers content scripts for newly authorized custom domains.

** 3. The Privileged Shell (Viewer)
- *Source*: =src/viewer.ts=
- *Output*: =viewer.js=
- *Environment*: Extension context (=chrome-extension://...=), full API access
- *Role*: Security gatekeeper and data acquisition layer.
- *Actions*:
  1. *Domain Validation*: Checks if the domain is authorized (default or custom).
  2. *Format Validation*: If unknown, shows a format selector.
  3. *SSRF Protection*: Blocks private IP ranges and loopback addresses.
  4. *File Fetching*: Downloads the structure file with a 25 MB size limit.
  5. *Settings Validation*: Merges storage settings with defaults, enforcing a strict allowlist of keys.
  6. *Sandbox Initialization*: Spawns the sandbox iframe and passes the file data and settings via =postMessage=.

** 4. The Engine (Sandbox)
- *Source*: =src/sandbox.ts=, =src/native-builder.ts=
- *Output*: =sandbox.js= (bundles both modules)
- *Environment*: Sandboxed iframe, origin =null=, no extension API access
- *Role*: Renders the structure using Mol*'s native engine.
- *Actions*:
  1. Signals readiness with =SANDBOX_READY=.
  2. Validates the =INIT_MOLSTAR= message.
  3. Converts the base64 data URI to a blob URL.
  4. Initializes the Mol* viewer and loads the structure.
  5. Renders the scene using =NativeBuilder=, which applies:
     - Global target configurations (proteins, ligands, etc.).
     - Custom rules (highlighting, labels, tooltips).
     - Camera settings.

* Build Pipeline

#+begin_src
src/*.ts  ──→  tsc (type checking)  ──→  (no file output)
         ──→  esbuild (bundling) ──→  dist/*.js
public/* ──→  assemble.js         ──→  dist/chrome/
dist/*.js                                dist/firefox/
#+end_src

- *tsc*: Performs type checking without emitting files.
- *esbuild*: Bundles each entry point and its dependencies into a single JS file.
- *assemble.js*: Copies the appropriate manifest and assets for each browser.

* Cross-Browser Compatibility

- *Chrome (MV3)*: Uses a service worker for the background script and the =sandbox= manifest key.
- *Firefox (MV2)*: Uses an event page for the background script and a permissive CSP to allow =unsafe-eval=.

The TypeScript source is identical for both targets; only the manifests differ.
