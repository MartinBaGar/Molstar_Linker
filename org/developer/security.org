* Security & Threat Mitigation

Mol* Linker employs a *defense-in-depth* strategy to protect user privacy, prevent malicious payload execution, and secure local networks.

** 1. Local-Only Data Processing

Mol* Linker processes all data locally in the user's browser:
- Files are fetched directly from the source (GitHub, GitLab, custom domains) into the browser's memory.
- Rendering happens entirely on the user's machine using Mol*'s native engine.
- No data is uploaded to external servers.
- Data is erased when the workspace tab is closed.

** 2. Sandbox Containment (Exploit Isolation)

Mol* requires =eval()= and =new Function()= to compile WebGL shaders dynamically. To mitigate risks:
- Mol* runs in a sandboxed iframe (=sandbox.html=).
- The sandbox has a =null= origin and no access to extension APIs, cookies, or the parent page.
- Even if a malicious payload were embedded in a structure file, it would execute in isolation and be discarded when the tab closes.

** 3. SSRF (Server-Side Request Forgery) Protection

Mol* Linker requests broad host permissions to bypass CORS for custom domains. To prevent abuse:
- Only =https://= URLs are allowed.
- Requests to =localhost=, =127.0.0.1=, and private IPv4 ranges (e.g., =192.168.x.x=, =10.x.x.x=) are blocked.
- This prevents the extension from being used as a proxy to attack local networks.

** 4. Secure Message Passing

Data flows securely between extension components:
- *Input Validation*: The viewer and sandbox validate file formats against a hardcoded allowlist (=pdb=, =mmcif=, =gro=, etc.).
- *Origin Locking*: The sandbox checks =event.origin= and rejects messages not from the extension's own =chrome-extension://= or =moz-extension://= URL.
- *Structured Messages*: All communication uses typed messages (e.g., =InitMolstarMessage=, =SandboxReadyMessage=).

** 5. Memory & Payload Limits

To prevent denial-of-service (DoS) attacks or browser crashes:
- *URL Length*: Content scripts reject URLs longer than 2,048 characters.
- *File Size*: The fetcher enforces a 25 MB limit, checking both the =Content-Length= header and the final blob size.
- *Rule Limits*: Custom rules are capped at 50 per configuration.
- *Text Limits*: Labels and tooltips are truncated to 256 and 512 characters, respectively.

** 6. Strict Content Security Policy (CSP)

The extension uses a layered CSP:
- *Extension Pages* (popup, options): Locked to =script-src 'self'= and =object-src 'none'=.
- *Sandbox*: Only the sandbox is granted =unsafe-eval=, isolating the risk to Mol*'s shader compilation.

** 7. Settings Validation

To prevent injection of malicious settings:
- Storage keys are validated against a strict allowlist.
- Custom rules are sanitized and capped.
- Camera JSON is parsed and validated before application.

** 8. Dynamic Permissions

Custom domains are authorized on a per-domain basis:
- Users must explicitly grant access to each domain.
- Permissions can be revoked at any time.
- Content scripts are registered dynamically only for authorized domains.
